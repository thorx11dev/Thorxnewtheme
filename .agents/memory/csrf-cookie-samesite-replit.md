---
name: csrf-cookie-samesite-replit
description: Why login/registration failed with CSRF_ERROR in Replit preview, and the double-submit cookie caveats (stale cookies, third-party cookie blocking) that made partial fixes insufficient.
---

Replit's preview pane embeds the app in a cross-site iframe (top-level document is a different site than the app's own domain). Browsers refuse to send `SameSite=Lax`/`Strict` cookies on subresource (fetch/XHR) requests in that context — only on top-level navigation. Any session/CSRF double-submit cookie set with `SameSite=Lax` will get set once but never sent back, breaking auth/CSRF silently inside the Replit preview (works fine in a normal top-level browser tab).

**Fix layer 1:** use `SameSite=None; Secure` for cookies that must round-trip through the preview iframe. Replit's proxy always terminates HTTPS, so `Secure` is safe even in dev.

**Gotcha A — stale cached cookie:** if a CSRF/double-submit cookie middleware only issues the cookie when `!req.cookies[name]` ("set once, skip if present"), a browser that already cached the *old* cookie (issued before the SameSite/Secure fix) keeps it forever — the server can't inspect a stored cookie's attributes, only its value. Fix: always reissue the cookie on every safe request, or version-bump the cookie name (e.g. `foo.v2`) to force a fresh cookie.

**Gotcha B — third-party cookie blocking (the deeper, complete fix):** even with `SameSite=None; Secure` correctly set, modern browsers (Chrome's third-party cookie phase-out, Firefox ETP strict, Safari ITP, Brave) still block storing/sending the cookie at all, because from the browser's point of view a cookie set by an iframed site whose origin differs from the top-level document is a *third-party cookie*, regardless of SameSite value. This is the actual reason `SameSite=None` alone doesn't fully fix auth inside Replit's preview iframe for many users.
**Real fix:** add `Partitioned: true` to the cookie options (both the CSRF cookie and the session cookie, e.g. `express-session`'s `cookie: { ..., partitioned: true }` and `res.cookie(name, val, { ..., partitioned: true })`). This is CHIPS (Cookies Having Independent Partitioned State) — partitioned cookies are exempted from third-party-cookie blocking because they're scoped per top-level site, so they still round-trip correctly inside an iframe even under strict blocking policies. Requires `cookie` package >=0.6 / `express-session` >=1.18 and MUST be combined with `SameSite=None; Secure` (partitioned cookies require Secure).
**How to apply:** whenever `sameSite === "none"` is chosen (i.e. cross-site cookie usage, which on Replit means "isReplit or prod"), also set `partitioned: true` on that same cookie. Verify via `curl -D -` that the `Set-Cookie` response header ends with `; Secure; SameSite=None; Partitioned`.
