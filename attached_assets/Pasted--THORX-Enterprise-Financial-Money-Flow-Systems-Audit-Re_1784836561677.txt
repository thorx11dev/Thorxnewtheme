### 🚨 Core Guidelines & Standards (Non-Negotiable)

1. Enterprise Precision: Har monetary calculation me floating-point arithmetic strictly banned hai. Har jagah `Decimal.js` aur DB level par `decimal(10,4)` / `decimal(10,2)` enforce karo.
2. Race Guard & Concurrency: High-traffic endpoints par `pg_advisory_xact_lock`, `SELECT ... FOR UPDATE`, aur partial unique indexes (`uniq_withdrawals_one_pending_per_user`, `uniq_user_transactions_source`) ki complete safety honi chahiye.
3. Atomic DB Transactions: Earnings, fee splits, aur balance deductions hamesha `db.transaction()` wrapper me atomic hone chahiye. Partial writes allowed nahi hain.
4. Clear Communication: Agar kisi bhi logic, flow, environment key, ya database schema me koi confusion, ambiguity, ya masla pesh aaye, toh bina kisi memory-based assumption ke **MUJH SE DIRECT SAWVAL POOCHO**. Assumptions bilkul nahi lagani.

### 📋 Detailed System Blueprint (In 9 Systems Ko Is Tarah Kam Karna Chahiye)

#### 1.1 TX-Point / Earnings Ledger System
- Gross PKR ko Engine-wise split karo (Engine A/B: 60% User / 40% Thorx, Engine C: 45% User / 55% Thorx).
- Dual-layer separation: Layer 1 (TX-Points) sirf gamified UI display ke liye ho, jabki Layer 2 (Real PKR) immutable withdrawal base bane.
- Points conversion (`CONVERSION_RATE`) aur rank-based variance `drawThorxCard()` se clean record ho.
- Idempotency guard (`uniq_user_transactions_source`) se duplicate ad views / tasks block hon.

#### 1.2 Ledger-Based Withdrawal Engine
- Minimum payout threshold (Rs. 100) enforce ho.
- Chronological FIFO scan (`calculateWithdrawalBreakdown`) unwithdrawn `user_transactions` ko consume kare.
- Partial last row handling ke liye `split_remainder` row generation exact ho.
- User balance debit aur transaction marking single atomic transaction me ho.

#### 1.3 Platform Fee Split System
- Gross withdrawal par 15% Platform Fee apply ho.
- Fee Split: 50% Referrer Commission (7.5% gross), 50% Thorx Net Profit (7.5% gross).
- Admin approval ke waqt dynamic rates aur net amounts (`net_amount`, `thorx_fee_share`, `referral_commission_paid`) DB me accurately persist hon.

#### 1.4 Referral Payout Engine
- Single-Tier (L1) direct referral commission model execute ho.
- Commission earn-time par nahi, बल्कि **Withdrawal Approval** time par trigger ho.
- Direct credit user ke Cash Wallet (`balanceCashPkr`) me jaye aur `referral_commissions` audit row create ho.

#### 1.5 Balance Management & Race Guards
- Four distinct wallet columns (`txPointsBalance`, `totalEarnings`, `availableBalance`, `balanceCashPkr`) ki isolation maintain ho.
- Parallel withdrawal requests aur manual adjustments ko `SELECT ... FOR UPDATE` se lock kiya jaye.
- Ad-view concurrency ko `pg_advisory_xact_lock(userId)` se serialize kiya jaye.

#### 1.6 Referral Cash Withdrawal System
- Dedicated endpoint `POST /api/withdrawals/referral` ka flow operational ho.
- Direct Cash Wallet (`balanceCashPkr`) se minimum Rs. 50 ki withdrawal validate aur process ho.
- Pure PKR transaction ho, isme FIFO scan ki zaroorat nahi.

#### 1.7 Withdrawal Preview Calculator
- Read-only simulation endpoint `GET /api/withdrawals/preview` bina state mutate kiye exact FIFO scan aur fee breakdown return kare.
- Client UI confirmation se pehle accurate prediction dictate kare.

#### 1.8 Withdrawal Idempotency Cache (In-Memory)
- `X-Idempotency-Key` header support ke sath 60-second TTL in-memory `Map` cache run kare.
- Har 30 seconds par automatic stale entry cleanup routine chale.
- Duplicate clicks par DB touch kiye bina cached response immediately serve ho.

#### 1.9 Founder Withdrawal Tracker
- Isolated `founder_withdrawals` table ke zariye platform net profit extraction manage ho.
- Permission `VIEW_PROFIT_LEDGER` strictly enforced ho.
- Profit Ledger formula: `(SUM(thorx_fee_share) + HilltopAds Balance) - SUM(founder_withdrawals)` bilkul accurate ho.
