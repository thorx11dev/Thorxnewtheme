Guild, Vault & Points Ledger — Implementation Blueprint
Key mechanics locked in
Withdrawal fee: single 15% fee total. A configurable REFERRAL_FEE_SHARE_PCT (e.g. 50%) of that 15% goes to the direct (L1) referrer; the rest stays with the platform. Total user deduction stays at 15%, not 25%+.
Existing pending L2 commissions: paid out once at their already-locked amount/rate, then L2 code path is retired for good.
Device lock: registration for role: "user" hard-blocks if a fingerprint is already bound to an existing user-role account (cap goes from today's 2→1). team/founder/admin stay exempt exactly as today, so one personal account + one team account on the same device remains fine.
Guild join: public searchable directory of open guilds; join = request, captain approves/rejects (uses guild_members.status = 'pending').
Weekly cycle boundary: fixed UTC week, Monday 00:00 UTC → Sunday 23:59:59 UTC (documented in code comments, not user-configurable in v1).
Historical backfill: skipped — points_ledger starts recording from deploy forward only; documented clearly in code and migration notes.
Phase 1 — Schema (shared/schema.ts)
guilds: name, captainId, guildRank (E–S), guildScore, strikes, status (active/frozen/disbanded), weekly target config snapshot, timestamps.
guild_members: guildId, userId (unique), status (pending/active/left), joinedAt, isCaptain flag.
guild_vault_ledger: per-guild-per-week accrual rows (guildId, weekStart, pointsAccrued, status).
guild_weekly_cycles: guildId, weekStart/weekEnd, target, goalMet, resolved flag, resolvedAt — unique per guild+week for idempotency.
points_ledger: userId, sourceType, pointsDisplayed, lockedPkrValue, conversionRateUsed, guildId (nullable), isConverted, createdAt.
Add personalRank/guildContributionScore to users (separate from existing rank/avatar system — no touching that).
New systemConfig keys with sane defaults: VAULT_HOLD_PCT (15), VAULT_MULTIPLIER_BY_RANK (E–S map), WEEKLY_GOAL_TARGETS_BY_RANK, CONVERSION_RATE (e.g. 100 pts = Rs 1), REFERRAL_FEE_SHARE_PCT (e.g. 50% of the 15% fee).
Push via executeSql (drizzle-kit push doesn't run non-interactively here, per prior convention).
Phase 2 — Points ledger + vault split (server/storage.ts, server/routes.ts)
Wherever ad-view/task completion currently credits PKR directly, insert a points_ledger row locking lockedPkrValue/conversionRateUsed at that instant; derive the wallet credit from that locked row, not a live rate.
Split 85/15 only when the user is an active, non-frozen guild member; otherwise 100% to instant wallet.
All of this inside db.transaction with .for('update') row locks, matching checkAndUpdateRank's pattern.
Phase 3 — Weekly resolution job + strikes
New job (setInterval/cron style, like leaderboard-cleanup.ts) that, per guild per completed UTC week: checks target met, applies multiplier or flat release, credits members, marks cycle resolved. Idempotent (checks resolved flag first).
Strike accrual (inactivity/fraud hooks, reusing the risk engine where sensible) + 3-strikes freeze blocking vault releases/accrual; admin add/clear-strike action.
Phase 4 — Referral simplification + device lock (server/storage.ts, server/routes.ts)
Rework createWithdrawal: single 15% fee, REFERRAL_FEE_SHARE_PCT of it credited to L1 referrer only; remove L2 lookup/insert entirely.
One-time migration step: pay out existing pending L2 commissionLogs rows at their locked amount, then mark handled.
Tighten device fingerprint check: user-role cap 2→1; team/founder/admin exemption unchanged.
Phase 5 — Backend APIs
Guild CRUD (create/search/join-request/approve/leave), guild status (vault, weekly progress, strikes, members), user's points-ledger history + running PKR total, admin moderation (strike/freeze/unfreeze/disband) and config endpoints — reusing existing auth/role middleware.
Phase 6 — Admin/Team Portal (client/src/components/admin/*)
SystemSettingsManager.tsx: new config section (vault %, per-rank targets/multipliers, conversion rate, referral share %) using the existing EconomicControl/save-mutation pattern.
New "Guilds" panel in AdminSidebar.tsx/AdminLayout.tsx: searchable guild table (rank, members, vault, strikes, status) + per-guild strike/freeze/disband actions, styled like UserManager.tsx/RiskWatchlistPanel.tsx.
PayoutControl.tsx: replace L1/L2 breakdown display with the single referral-share line.
TaskManager.tsx: light-touch visibility of upcoming/failed weekly resolutions only (no duplicate moderation UI).
Phase 7 — User Portal (client/src/pages/UserPortal.tsx)
Scratch Card reveal component replacing the plain success toast after ad/task completion, driven by the real points_ledger row from the API response.
New "Guild" nav section: no-guild state (search/create/join-request), guild state (vault balance, weekly progress bar, members + captain badge, strikes, countdown to resolution).
Ledger/wallet view: TX-Points balance + live PKR equivalent, paginated points_ledger history.
Referrals section: strip Level-2 references, show single-level share only.
All wired through existing apiRequest/React Query conventions already used in the file.
Phase 8 — Verification
Manual test pass: earn → ledger row → vault split → weekly resolution (goal met & failed cases) → withdrawal with new fee/referral math → device-lock rejection → guild join/approve flow, all against the real HTTPS dev domain per project convention.
Confirm existing rank/avatar system, risk engine, and withdrawal atomicity guarantees are untouched.
