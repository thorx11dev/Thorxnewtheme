PKR Leaks in Frontend

Several components are exposing PKR values directly to users: the withdrawal submission and preview screens both display net amounts in rupees, the detailed breakdown shows the full rupee calculation, and the profile modal mislabels a PKR-based field as "TX-Points". The terms also explicitly reference PKR as a real currency rather than an abstraction.

DB Transactions:
createReferral lacks atomicity with just a single insert and no transaction wrapper, while processWithdrawal depends on the lock from createWithdrawal rather than acquiring its own lock on the users row.

Race conditions:
There's no reserved points mechanism between createWithdrawal and processWithdrawal, leaving pending withdrawals vulnerable to concurrent operations.

Cascade Delete:
The points_ledger uses CASCADE deletion when it should use RESTRICT since it's an audit trail, and ad_views also cascades when it shouldn't.

Identity fields lack uniqueness constraints — both users.identity and users.phone are missing UNIQUE indexes, creating multi-account abuse vectors.
CATEGORY 2 — Million Dollar Standards:

Auth:
The PATCH /api/users/:id endpoint bypasses the suspension check by manually calling getThorxPrincipalId instead of using requireSessionAuth middleware, and the health check doesn't verify database connectivity. Rate limiting relies on in-memory storage rather than Redis, which breaks across multiple instances.

Input Validation:
Most routes have Zod schemas in place, though I need to verify the complete coverage.

Missing DB Indexes:
The notifications table is missing a composite index on user_id and is_read, and the score table needs indexing as well.

Several tables lack composite indexes that would improve query performance — score_history needs user_id plus snapshot_at, chat_messages needs guild_id plus created_at, ad_views needs user_id plus created_at, and points_ledger has separate indexes but is missing the user_id plus created_at composite.

Testing:
There's only one test file covering financial logic, with no frontend, E2E, or integration tests.

Logging:
Console.log statements are scattered throughout the codebase — in the scheduler, health engine, validation module, and Vite config — plus the debug utility itself uses console.log internally.

Sentry integration exists only on the server side with no frontend coverage.

Health check...

Ecosystem & UX issues:

The admin dashboard shows ellipsis during loading instead of skeleton screens, the referral section uses a basic spinner, the engine selector tab has no loading state, and rank refresh silently fails. Mobile needs attention too — there are console.log issues in admin tables and the team portal needs a mobile check.
