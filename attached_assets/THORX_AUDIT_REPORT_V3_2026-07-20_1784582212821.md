## CATEGORY 1 — Mistakes, Bugs & Gaps

---

### 1.1 — Database Integrity

---

**Finding 1-A — `createReferral()` Is Not Atomic With `createUser()`** `HIGH`
- **File:** `server/storage.ts` lines 697–710 (createUser), 1177–1179 (createReferral)
- **Code:**
  ```typescript
  // createUser() — two separate DB operations, no transaction:
  const [user] = await db.insert(users).values(userData).returning();   // committed
  // ...
  await this.createReferral({ referrerId, referredId: user.id, ... });  // separate insert
  ```
- **Issue:** The user row is committed before the referral insert runs. If `createReferral()` fails (DB timeout, constraint violation, network blip), the user is created but their referral relationship is permanently lost. The referrer never earns commission on this user, and the platform silently loses referral revenue. This is a TOCTOU gap in the registration flow.
- **Fix:** Wrap both inserts in `db.transaction()` inside `createUser()`. If the referral insert fails, the entire registration rolls back and the user retries.

---

**Finding 1-B — `users.identity` and `users.phone` Missing UNIQUE Constraints** `HIGH`
- **File:** `shared/schema.ts` lines 26–27
- **Code:**
  ```typescript
  identity: text("identity").notNull(),   // ← no unique()
  phone: text("phone").notNull(),         // ← no unique()
  ```
- **Issue:** `identity` is the user's government-issued ID (CNIC equivalent) — a unique national identifier. Without a UNIQUE constraint, a single person can create unlimited accounts with the same identity document. Same for `phone`. The app does uniqueness checking at the application layer only — a race condition between two concurrent registrations with the same identity bypasses even the app-level check. Multi-account fraud is trivially enabled.
- **Fix:** Add `unique()` to both columns. Add a DB-level partial unique index for `phone` to exclude empty strings (many users may have `phone = ""`):
  ```sql
  CREATE UNIQUE INDEX IF NOT EXISTS users_identity_unique ON users(identity) WHERE identity != '';
  CREATE UNIQUE INDEX IF NOT EXISTS users_phone_unique ON users(phone) WHERE phone != '';
  ```

---

**Finding 1-C — `onDelete: "cascade"` on Financial & Audit Tables** `HIGH`
- **File:** `shared/schema.ts`
  | Line | Table | Column | Current | Risk |
  |------|-------|--------|---------|------|
  | 185 | `ad_views` | `user_id` | cascade | Earn history deleted |
  | 306 | `task_records` | `user_id` | cascade | Task completion history deleted |
  | 1173 | `points_ledger` | `user_id` | cascade | Financial ledger deleted |
  | 532 | `risk_cases` | `user_id` | cascade | Fraud audit trail deleted |
- **Issue:** Any admin "delete user" operation (or a mistaken `DELETE FROM users WHERE ...`) permanently destroys the user's entire earn history, task records, financial ledger, and fraud investigation trail. Standard financial record-keeping requires retaining transaction data even after user deletion (7-year rule in most jurisdictions). Soft-delete is the correct pattern.
- **Fix:** Change `onDelete: "cascade"` → `onDelete: "restrict"` on all four tables. Implement soft-delete for users (`isActive = false`, email anonymized). The admin "terminate node" action already sets `isActive: false` — just remove the hard-delete code path.

---

**Finding 1-D — `admin/users/:id/action` Balance Adjust Has No Audit Log** `HIGH`
- **File:** `server/routes.ts` lines 500–515
- **Code:**
  ```typescript
  } else if (action === "adjust_balance" && payload && payload.amount) {
    const amount = new Decimal(String(payload.amount));
    const currentBalance = new Decimal(user.availableBalance || "0");
    await storage.updateUser(id, { availableBalance: currentBalance.plus(amount).toFixed(4) } as any);
    // ← no storage.createAuditLog() call
  }
  ```
- **Issue:** The admin action route bypasses `storage.adjustUserBalance()` (which creates audit log entries) and calls `storage.updateUser()` directly. Every balance change through the admin panel is invisible in the audit log. A dishonest admin can add unlimited funds to any account with no paper trail.
- **Fix:** Replace `storage.updateUser()` call with `storage.adjustUserBalance(id, amount, adminId, reason)` which already creates an audit log entry.

### 1.2 — Authentication & Authorization

---

**Finding 1-E — `PATCH /api/users/:id` Bypasses Suspension Check** `HIGH`
- **File:** `server/routes.ts` lines 677–680
- **Code:**
  ```typescript
  app.patch("/api/users/:id", profileRateLimiter, async (req, res) => {
    const principalId = getThorxPrincipalId(req);   // ← manual session extraction
    if (!principalId) return res.status(401)...;
    // No check for user.isActive — suspended users can still update their profile
  ```
- **Issue:** The route uses `getThorxPrincipalId(req)` instead of the `requireSessionAuth` middleware. The middleware checks `user.isActive` and rejects suspended users. The manual check only validates session existence — a suspended user who still has a valid session cookie can fully update their profile.
- **Fix:** Replace manual session check with `requireSessionAuth` middleware, and verify `user.isActive` before processing updates.

---

**Finding 1-F — Suspended User WS Session Not Immediately Invalidated** `MEDIUM`
- **File:** `server/routes.ts` line 511 (suspend action), `server/realtime.ts`
- **Issue:** When an admin suspends a user (`isActive: false`), the user's existing WebSocket connection is not terminated. The user continues to receive real-time updates, guild messages, and balance notifications for the lifetime of their current WS session (up to the server restart or manual disconnect). Suspension takes effect only on the next HTTP request, not in real-time.
- **Fix:** After setting `isActive: false`, iterate `sockets` in `realtime.ts` to find connections belonging to this `userId` and close them with code `4003` ("Account suspended").

### 1.3 — Floating-Point Precision

---

**Finding 1-G — `parseFloat` in Risk-Engine Scoring** `MEDIUM`
- **File:** `server/modules/risk-engine.ts`
  - Line 66: `const earned24h = parseFloat(row?.total ?? "0")`
  - Line 290: `const totalEarnings = parseFloat(user?.totalEarnings ?? "0")`
- **Issue:** Risk scoring uses native float for earnings comparisons. While risk scores don't move money directly, a float drift error in `earned24h` could cause a legitimate high-earning user to be incorrectly flagged (false positive) or a fraudster to evade detection (false negative). The pattern also contradicts the project's Decimal-everywhere policy.
- **Fix:** `new Decimal(row?.total ?? "0").toNumber()` — consistent with the pattern used everywhere else.

### 1.4 — Frontend Debug Code

---

**Finding 1-H — `console.log/error` in `health-engine.ts` Not DEV-Gated** `MEDIUM`
- **File:** `server/modules/health-engine.ts` lines 633, 635
- **Code:**
  ```typescript
  console.log(`[HealthEngine] Snapshot saved — overall: ${result.overallScore}, reason: ${result.topReason.slice(0, 80)}`);
  // ...
  console.error("[HealthEngine] Failed to compute/save snapshot:", err);
  ```
- **Issue:** Two unstructured `console.*` calls remain in the health engine. Line 633 logs success status to stdout as an unstructured string (not captured by pino). Line 635 logs full error detail (including potential DB error context) as plain text. Both bypass the structured pino logger used everywhere else in the server.
- **Fix:** Replace both with `logger.info(...)` and `logger.error({ err }, ...)` using the existing pino logger import.

## CATEGORY 2 — Million-Dollar Company Standards Gaps

---

### 2.1 — Infrastructure

---

**Finding 2-A — `/api/health` Returns `healthy` Without Checking Database** `HIGH`
- **File:** `server/routes.ts` line 366–369
- **Code:**
  ```typescript
  app.get("/api/health", (_req, res) => {
    res.status(200).json({ status: "healthy", timestamp: new Date().toISOString() });
  });
  ```
- **Issue:** The health check endpoint always returns `200 healthy` — even if the database is completely unreachable. Any health-check-based deployment tool (Kubernetes liveness probe, load balancer health check, uptime monitor) will consider the app healthy when it is actually unable to serve any data-dependent request. A silent DB connection failure goes undetected until users start reporting errors.
- **Fix:** Run a lightweight `SELECT 1` query against the DB; return `503` if it fails:
  ```typescript
  app.get("/api/health", async (_req, res) => {
    try {
      await db.execute(sql`SELECT 1`);
      res.json({ status: "healthy", db: "connected", timestamp: new Date().toISOString() });
    } catch (err) {
      res.status(503).json({ status: "unhealthy", db: "disconnected", timestamp: new Date().toISOString() });
    }
  });
  ```

### 2.2 — Performance

---

**Finding 2-B — `refreshLeaderboardCache()` Still Loads 10,000 Users Into Node Heap** `MEDIUM`
- **File:** `server/storage.ts` lines 3020–3100
- **Issue:** The V2 fix capped the memory bomb from "all users" to `TOP_N = 10,000`. At 10,000 users, the function still fetches 10k rows into Node.js heap, performs in-memory Decimal-comparison sort, then writes back in 500-row chunks. A pure SQL `RANK() OVER (ORDER BY performance_score DESC)` window function computes ranks inside the database with zero heap allocation. The V2 plan specified this approach but the implementation took a shortcut.
- **Fix:** Compute rank positions entirely in SQL using `RANK() OVER`. Only fetch the computed results (top 200 for display) back to Node.js.

---

**Finding 2-C — Missing Composite Index `notifications(user_id, is_read)`** `MEDIUM`
- **File:** `shared/schema.ts` lines 524–526
- **Issue:** Three individual indexes exist on `notifications` (user_id, type, created_at). The most common query — "get unread notifications for user X" — filters on `(user_id = ? AND is_read = false)`. This requires a composite index; the current individual `user_id` index still requires an in-memory filter for `is_read`. At 10k+ notifications per user this becomes a noticeable query.
- **Fix:** `index("notifications_user_id_is_read_idx").on(table.userId, table.isRead)` in schema.

---

### 2.3 — Observability

---

**Finding 2-D — No Frontend Error Tracking (Sentry)** `MEDIUM`
- **Issue:** Backend has Sentry integrated (Task 21 from V2). Frontend has zero error tracking. React component crashes caught by `ErrorBoundary` are only logged to the browser console in DEV mode — in production, a user hitting a crash sees a blank error page with no alert sent to the team. Client-side JS errors (failed fetches, broken renders, type errors) are completely invisible to the engineering team.
- **Fix:** Initialize `@sentry/react` in `client/src/main.tsx`. Gate on a `VITE_SENTRY_DSN` env var. Wrap the React tree with `Sentry.ErrorBoundary` as fallback to the existing `ErrorBoundary`.

## CATEGORY 3 — UX Friction & Ecosystem Gaps

---

### 3.1 — Loading State Gaps

---

**Finding 3-A — `ProtectedRoute` Returns `null` on All 3 Guard Variants** `HIGH`
- **File:** `client/src/components/auth/ProtectedRoute.tsx` lines 34, 66, 116
- **Code:**
  ```tsx
  if (isLoading) return null;   // ← line 34 (UserRoute)
  if (isLoading) return null;   // ← line 66 (GuestRoute)
  if (isLoading) return null;   // ← line 116 (TeamRoute)
  ```
- **Issue:** Every protected page flashes a blank white screen for ~100–300ms on every page load while the session check resolves. On mobile or slow connections this flash is 500ms+. Users on first load see: blank → redirect, or blank → content. Both look broken. This is the single most visible "cheap app" signal on the platform.
- **Fix:** Return a full-page skeleton that matches the route's layout (nav skeleton + content area pulse) during `isLoading`. This eliminates the visual flash entirely.

---

### 3.2 — Real-Time Feedback

---

**Finding 3-B — WS Disconnect Has No Persistent Status Indicator** `MEDIUM`
- **File:** `client/src/hooks/useRealtimeSync.ts` line 243
- **Issue:** When the WebSocket disconnects, a one-time toast fires ("Real-time features paused — reconnecting…") then disappears. If the user doesn't see the toast (looking away, phone locked), they have no way to know real-time features are down. They can continue typing chat messages that appear sent locally but are not delivered. A persistent visual indicator (subtle badge or banner) should remain until reconnection is confirmed.
- **Fix:** Expose `wsConnected: boolean` from `useRealtimeSync`. Show a persistent bottom banner while disconnected. Remove it immediately on reconnect.
