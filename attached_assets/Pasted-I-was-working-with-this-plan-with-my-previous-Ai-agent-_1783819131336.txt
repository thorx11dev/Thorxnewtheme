Here's a strategic teardown of both systems as they stand today, and what it would take to bring them to institutional-grade quality.

Where they actually stand today
Scoring is a single static formula (earnings 40% / team 30% / activity 15% / health 15%), recomputed only when an admin clicks a button, with no history and no way to tune weights without a code change.

Watchlist is the bigger gap. It's not really a watchlist — it's two if statements evaluated fresh on every page load: "earned >$5k in <24h" and ">50 referrals but <$100 earned." Nothing persists. There's no case status, no assignment, no audit trail, and no memory — an admin who clears a flagged account will see it flagged again the moment they refresh, because there's nowhere to record "we looked at this, it's fine."

That's the core problem worth fixing first: a watchlist that can't remember its own decisions isn't a watchlist, it's a filter.

Strategic plan
1. Split "merit" from "risk" — they're different questions
Right now the Performance Score quietly blends how well someone is doing with how suspicious they look (bot-network ratio lives inside the same number space conceptually). A $10k/month company can't make good decisions this way — a legitimate power-user and a fraud ring can produce similar-looking spikes. Two independent scores:

Performance Score (existing, refined) — merit, drives rank/leaderboard.
Risk Score (new, replaces the two hardcoded rules) — continuous 0–100, drives the Watchlist.
2. Scoring upgrades
Percentile/z-score normalization instead of max-normalization. Right now everyone's earnings score is you / topEarner. One whale account silently compresses everyone else's score toward zero. Percentile rank is far harder to game and far more stable.
Score history table — snapshot scores on each recompute so admins see trend, not just a point-in-time number. A score that jumped 40 points overnight is itself a signal.
Cohort-aware scoring — a 3-day-old account shouldn't be judged on the same curve as a 6-month veteran; segment or apply a confidence discount for sparse data.
Admin-tunable weights stored in system config (you already have this pattern for payout rules) instead of hardcoded constants, with an audit log entry every time weights change.
3. Watchlist upgrades — this is where the real value is
Turn it from a stateless filter into an actual case-management system:

Persist it. A risk_cases table: user, risk score, triggering signals, status (Open → Investigating → Cleared/Actioned), assigned admin, notes, timestamps. This alone fixes the "keeps re-flagging cleared accounts" problem.
Close the loop with Trust Status. You already built Trust Status (Special/Trusted/Normal/Dangerous) — right now it's disconnected from the watchlist. Trust Status should be the resolution of a risk case: an admin investigates, then sets the outcome, and it's logged as the answer to "why is this account marked Dangerous."
Expand signals beyond the two current heuristics (ranked by fraud-detection ROI):
Device/IP fingerprint clustering across accounts (you already collect deviceFingerprint at registration — it's underused)
Referral tree shape anomalies — long linear chains rather than organic branching is a classic pyramid-scheme signature
Cash-out velocity — withdrawal immediately after earnings post, before any organic delay
Circular/self-referrals
Task-completion speed faster than physically plausible
Severity tiers, not booleans — Low/Medium/High/Critical, so admins triage by urgency instead of an undifferentiated list.
Configurable thresholds — move the $5,000/24h and 50-referral/$100 numbers into system config so fraud patterns can be retuned without a deploy.
Push alerts, not polling — you already have a WebSocket realtime layer for rank-ups; reuse it so a Critical-severity case pings admins the moment it's detected instead of waiting for someone to open the tab.
Feedback loop — log which flags admins confirm as real fraud vs. dismiss as false positives. Over time this tells you which signals are actually predictive and which are just noise, so the rules get sharper instead of static forever.
