## 1. THE ULTIMATE ILLUSION & BACKEND LEDGER (Engine A, B & C)

The user must NEVER see or know their real PKR balance anywhere in the application except at the final step of the payout/withdrawal preview. Every dashboard, live feed, card reward, and user-facing portal must only display **TX-Points** (referred to as "the illusion").

### 1.1 — Team Portal Configuration (The Control Board)
In the Team Portal (Admin/Staff), provide fields to configure each ad network/engine:
* **Engine A (Ad Player Specific)**: Every individual ad player slot must have its own field to configure:
  1. `pkr_to_tx_points_ratio` (e.g., 1 PKR = 1000 TX-Points)[span_1](start_span)[span_1](end_span).
  2. `illusion_variance_percentage` (e.g., 5% or 10%). This controls how close to the average points the random Thorx Card rewards should be shown.
* **Engine B**: Same fields as above.
* **Engine C**: Same fields as above.

### 1.2 — Earning Flow (The 40/60 Rule & Thorx Cards)
When a user completes an ad, task, or CPA offer:
1. **Gross Revenue Received**: Let's say `$REVENUE` in PKR is recorded from the network.
2. **Platform Cut (40/60 Rule)**:
   * **40%** goes directly to the **Thorx Profit Card** (stored in the database and visible only in the Team Portal Finance Section).
   * **60%** is the user's Net PKR Share.
3. **The Backend Ledger**: This 60% Net PKR is immediately added to the user's `users.real_pkr_balance` (securely stored in the database, hidden from the user).
4. **The Frontend Thorx Card (The Illusion)**:
   * Calculate the average TX-Points based on the network's configured `pkr_to_tx_points_ratio` for that specific slot.
   * Apply the `illusion_variance_percentage` to generate a random points amount that is **extremely close to the average** (no massive gaps or weird wild numbers).
   * Display this calculated TX-Points reward on the **Thorx Card** animation. The user collects these points, believing they correspond to a mysterious float value.

### 1.3 — Manual Balance Adjustment (Credit/Debit Guardrail)
When a Team Member manually adds or removes balance for a user from the Team Portal:
1. The Staff member must manually type TWO separate fields:
   * **Field 1**: The Real PKR Value to add or deduct from the database (`real_pkr_balance`).
   * **Field 2**: The TX-Points Value to add or deduct from the user's dashboard (to maintain the illusion).
2. The transaction only completes when both fields are entered and confirmed. This ensures the database ledger matches the visual points the user sees on their screen.

## 3. ADVANCED REFERRAL FEES & THORX PROFIT LEDGER

### 3.1 — Non-Destructive Referral Shares
When User A invites User B (via code or link):
1. User B signs up and earns points normally.
2. When User B requests a **Withdrawal**, a **15% Platform Fee** is deducted from User B's payout.
3. **The Referral Cut**:
   * In the Team Portal, Staff can configure a global or per-user `referral_fee_share_percentage` (e.g., 50%).
   * **Example**: If User B's 15% fee equals `Rs. 100`:
     * **50% of the fee (Rs. 50)** is instantly credited to **User A's** `real_pkr_balance` (with a proportional, configurable TX-Points illusion adjustment applied to User A's dashboard so their points increase smoothly).
     * **The remaining 50% (Rs. 50)** goes directly to Thorx's company pocket.

### 3.2 — Thorx Profit Card (Team Portal Finance Section)
Create a clean, dedicated dashboard card/ledger in the Team Portal called **Thorx Profit Card**. This card must accumulate:
* **The 40% Platform Cut** from all completed tasks, ads, and Engine A/B/C earnings.
* **The Platform's remaining share of the 15% Withdrawal Fee** after paying out any active referral commissions.
