#3 - Guild Vault & Points Ledger Backend
What & Why
Implement the foundational backend/database layer for the Thorx Master Plan (thorx_master_plan.md): Engine C Guilds, the 15% "Hold & Release" Escrow Vault, the simplified 1-level referral share, and the bulletproof points_ledger valuation system. This is the data + business-logic layer that the User Portal and Team/Admin Portal tasks will build on top of, so it must land first.

Today the platform pays users directly in PKR decimals (users.availableBalance etc.) with no points illusion layer, referral commissions are split across two levels (L1_BONUS/L2_BONUS read in server/storage.ts around processWithdrawal/createWithdrawal), and there is no guild/team concept at all. This task migrates that model to points-first accounting with a locked-PKR ledger, without breaking the existing atomic withdrawal guarantees (row locks, pending-withdrawal checks, balance constraints) that already exist in createWithdrawal/processWithdrawal/rejectWithdrawal.

Done looks like
New tables exist for guilds, guild membership, guild vault ledgers, weekly guild cycles, and points_ledger, wired into shared/schema.ts with Drizzle relations and Zod insert schemas following existing conventions.
Every point-earning action (ad view, CPA/daily task completion) writes a points_ledger row that hard-locks locked_pkr_value and conversion_rate_used at that exact moment, and that value is what increments the user's spendable PKR balance — future admin changes to the conversion rate never retroactively change already-earned value.
Each qualifying earn event automatically splits 85% to the user's instant wallet and 15% into their guild's vault (only when the user belongs to an active, non-frozen guild; if the user has no guild, 100% goes to the instant wallet).
A weekly job resolves each guild's vault: if the guild's weekly point target was met, frozen vault points are released to members with the rank-based multiplier (1.2x/1.5x etc., configurable); if not met, the same points are released flat with no multiplier. The job is idempotent per guild per week (safe to re-run/retry without double-paying).
Guilds have a working Strike system: 3 strikes freezes payouts (vault releases and/or new vault accrual) for that guild until an admin or cooldown clears it.
Withdrawal fee logic is simplified to a single referral level: on withdrawal, a configurable share of the 15% withdrawal fee is credited to the direct (Level 1) referrer only; all Level 2 commission code paths are removed or safely no-op'd, with existing pending L2 commissions handled gracefully (paid out or voided per a documented decision, not silently orphaned).
Device-ID lock: registration enforces a hard one-account-per-device rule (not just per-user uniqueness as today), rejecting or flagging signups that reuse a device fingerprint already tied to an existing account.
New REST endpoints exist for: creating/joining/leaving a guild, viewing guild vault status and weekly progress, viewing a user's points_ledger history and running PKR total, and admin endpoints for guild moderation (strike/freeze/unfreeze/disband) and vault-rate configuration — all protected by the existing auth/role middleware patterns used elsewhere in server/routes.ts.
All new money-moving logic runs inside db.transaction(...) blocks with row-level locking (for('update')) matching the pattern already used in checkAndUpdateRank/createWithdrawal, so concurrent ad completions, weekly releases, and withdrawals can't race each other or double-spend.
Out of scope
Any frontend/UI work (covered by the User Portal and Team/Admin Portal tasks).
Redesigning the existing five-tier named user rank system (Nawa Aya → Chacha Supreme) — Guild rank (E–S) is a separate, additional axis and must not overwrite or rename the existing users.rank field or its color conventions.
Payment gateway / real bank payout integration changes.
Historical backfill of existing earnings/adViews rows into points_ledger (new rows only, going forward) unless trivially cheap — if skipped, document it clearly.
Steps
Schema additions — Add guilds, guild_members, guild_vault_ledger (or equivalent per-guild-per-week accrual rows), guild_weekly_cycles (tracks target, start/end, resolved flag per guild per week), and points_ledger tables to shared/schema.ts, with indexes, FKs, Drizzle relations, and createInsertSchema exports following the exact style of existing tables (e.g. riskCases, commissionLogs). Add guildRank, guildScore, strikes to guilds and a per-user guildContributionScore/personalRank (E–S) field distinct from the existing named rank column. Push schema via the project's db:push flow (raw SQL via executeSql if drizzle-kit push can't run non-interactively, per prior project convention).
Points ledger write path — Wherever points/PKR are currently credited for an earn event (ad-view completion, daily task completion in server/routes.ts/server/storage.ts), insert a points_ledger row with the locked PKR value and conversion rate at that instant, and derive the credited wallet amount from that locked value instead of a hardcoded decimal, preserving the existing totalEarnings/availableBalance/pendingBalance update pattern.
Escrow vault split logic — On each qualifying earn event, branch on whether the user is an active guild member in good standing (not frozen): split the earned value 85/15, credit the vault ledger for the guild, and record which weekly cycle the 15% belongs to.
Weekly resolution job — Build a scheduled job (following the server/jobs/leaderboard-cleanup.ts setInterval pattern, or a proper cron if more appropriate for weekly cadence) that, per guild per completed week, atomically checks the goal, applies the multiplier or flat release, credits members' instant wallets, and marks the cycle resolved so re-runs are no-ops. Handle timezone consistently (fixed UTC week boundary) and document that choice.
Strike system — Implement strike accrual (inactivity / fraud signals — reuse or hook into the existing risk engine where sensible) and the 3-strikes freeze that blocks vault releases and/or new accrual for a guild, plus an admin action to add/clear strikes.
Referral simplification — Update createWithdrawal/processWithdrawal/rejectWithdrawal in server/storage.ts to compute a single Level-1 referral share from the withdrawal fee instead of L1+L2, remove the L2 beneficiary lookup and commission insert, and decide/implement handling for any already-pending L2 commissionLogs rows (pay at last-known rate or void with a clear audit note — pick one and document it in code comments).
Device lock hardening — Extend the registration flow to check deviceFingerprints (or a new lock table) for a fingerprint hash already bound to a different user and reject/flag that signup, closing the current per-user-only uniqueness gap.
Guild & ledger APIs — Add REST endpoints for guild CRUD (create/join/leave/list), guild status (vault balance, weekly progress, strikes), user ledger history, and admin moderation/config endpoints (strike, freeze, disband, set vault %, set multipliers, set conversion rate), reusing the existing auth/role-check and admin-permission patterns already present in server/routes.ts.
Config keys — Register new tunable systemConfig keys (e.g. VAULT_HOLD_PCT, VAULT_MULTIPLIER_BY_RANK, WEEKLY_GOAL_TARGETS_BY_RANK, CONVERSION_RATE, REFERRAL_FEE_SHARE_PCT) with sane defaults matching the master plan's example numbers, read via the existing getSystemConfigValue helper.
